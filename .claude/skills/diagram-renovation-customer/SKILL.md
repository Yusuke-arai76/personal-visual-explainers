# Diagram Renovation Customer

**窓・ドアのリフォームに興味を持ったお客様**が、補助金・助成金の仕組みをひとりで読んで腹落ちできるHTML図解を作る。

## 想定読者・成功定義（ヒアリング反映）

| 項目 | 内容 |
|------|------|
| 作者の主戦場 | 窓・ドアリフォーム業の**営業** |
| 読者 | **一般のお客様**。リフォームには興味があるが、補助金・助成金の制度は知らない |
| 典型ニーズ | 国・都道府県などの**行政主体の助成金**。文章の羅列で理解しにくい |
| 成功 | **まったく知識のない人が**、図解付き説明書を読んで、**自分で理解・納得できる**状態 |

> 「営業担当がいなくても読める」「手元に置いて後で見返せる」をゴールにする。

## 依存（このリポジトリ内）

- `.claude/skills/creating-visual-explainers/references/base.html` — 図解の**額縁**（Tailwind・Lucide・ADS配色・プレースホルダー）
- `.claude/skills/creating-visual-explainers/references/model-answer.html` — 品質・パターン参照
- `.claude/skills/creating-visual-explainers/scripts/deploy-diagram.sh` — surge.sh デプロイ
- `output/` — 生成HTMLの保存先

## ワークフロー

### Phase 0: リサーチ（サブエージェント）

**explore** サブエージェントで、制度の一次情報を取得して正確な事実を集める。

```
Task({
  subagent_type: "explore",
  description: "補助金・助成金の一次情報収集",
  prompt: `
    以下について徹底的に調査してください：
    【対象】{URL or 制度名（例：先進的窓リノベ2025事業、東京都環境局 助成金など）}

    調査項目：
    1. 制度の目的・背景（なぜ存在するか）
    2. お客様（個人・法人）が対象になる条件
    3. いくらもらえるか（上限・割合・計算方法）
    4. 手続きの流れ（申請〜入金のステップ）
    5. 期限・予算・注意事項
    6. よくある誤解・落とし穴
    7. 公式一次情報のURL

    結果は箇条書きで整理して返してください。
  `
})
```

### Phase 1: ソース読み込み

1. URLなら `WebFetch` で公式ページを取得
2. **数値・条件・期限は一次情報と照合**して確認（思い込みで書かない）
3. Phase 0 と突き合わせて抜け漏れを潰す

### Phase 2: お客様目線の疑問リストを作る

お客様が「読む前に持っている疑問」を想定する。

| よくある疑問 | 図解でどう答えるか |
|-------------|------------------|
| 自分の家は対象になる？ | 条件チェックリスト |
| いくら戻ってくる？ | 金額の計算例 |
| 何をすればいい？ | ステップフロー |
| 急いだ方がいい？ | 期限・残予算の見せ方 |
| 詐欺じゃない？ | 公式URLと行政名を明記 |

### Phase 3: 用語をお客様語に翻訳

技術語・行政語を一切使わず書ける言葉に変換する。

**詳細** → [references/term-dictionary.md](references/term-dictionary.md)

### Phase 4: 情報を絞り込む

- **「最初に伝える3つ」** を決定（対象・金額・手続きの流れ）
- お客様が自分で判断できる情報を優先する
- 例外・細目は後半の「詳しく知りたい方へ」に退避

### Phase 5: 安心感の設計

お客様は「難しそう」「怪しくない？」という不安を持っている。安心してもらうための工夫を盛り込む。

**詳細** → [references/character-usage.md](references/character-usage.md)

### Phase 6: HTML生成

1. `creating-visual-explainers/references/base.html` を `output/{スラッグ}.html` にコピー
2. プレースホルダーを置換（`<!-- TITLE -->` / `<!-- DESCRIPTION -->` / `<!-- CONTENT_START -->`〜`<!-- CONTENT_END -->`）
3. Tailwind（`ads.*` 配色）と **Lucide のみ**（絵文字禁止）
4. **「金額の計算例」は数値つきで**具体的に書く（「最大〇〇円」が伝わる）
5. 一次情報URLを必ず末尾に記載
6. **印刷・PDF想定のレイアウト**（A4縦で崩れないか意識する）

**詳細** → [references/html-structure.md](references/html-structure.md)

### Phase 7: 批判的レビュー（サブエージェント）

**補助金を全く知らないお客様の視点**で、忖度なくレビューする。

```
Task({
  subagent_type: "generalPurpose",
  description: "お客様視点での図解レビュー",
  readonly: true,
  prompt: `
    あなたは「窓のリフォームに興味があるが、補助金は全く知らない50代の専業主婦」です。
    スマホは使えますが、行政の手続きは苦手で、お金の話は慎重です。

    以下のHTML図解を読んで、忖度なく批判してください：

    【HTMLファイルパス】{path}

    ## レビュー観点

    1. **わからない言葉・略語**: 読んでいてつまずいた言葉はどれか
    2. **信頼できるか**: 「本当に大丈夫？」「怪しくない？」と感じた箇所
    3. **自分ごとになれるか**: 「これ、うちの家のこと？」と思えるか
    4. **金額が腹落ちするか**: 「実際にいくら得するか」がイメージできるか
    5. **次のアクションがわかるか**: 読み終えて「次に何をすればいいか」がわかるか
    6. **情報量**: 読むのが疲れる、または逆に足りない部分はどこか

    ## 出力形式

    ### 致命的な問題（これだと読み飛ばす）
    - ...

    ### 改善すべき点
    - ...

    ### 良かった点
    - ...

    ### 総評
    5段階評価と一言
  `
})
```

### Phase 8: ブラッシュアップ

Phase 7 の結果を反映して修正する。

**優先度**: 致命的 → 改善 → 良かった点は維持

**よくある修正パターン**:
- 「申請」「交付」などの行政語を日常語に差し替え
- 金額に具体的な計算例を追加
- 「公式サイトはこちら」リンクを目立つ場所へ移動
- チェックリストの条件表現を「はい/いいえ」形式にする

### Phase 9: デプロイ

```bash
node --version
bash .claude/skills/creating-visual-explainers/scripts/deploy-diagram.sh output/{スラッグ}.html [スラッグ]
```

または GitHub に push して GitHub Pages で公開。

## 品質チェックリスト

### 必須

- [ ] 行政語・専門語がすべてお客様語に変換されている
- [ ] 「自分の家が対象かどうか」の判断材料がある（条件チェックリストなど）
- [ ] 金額が**具体的な数字で**わかる（「最大〇〇万円」「工事費の〇〇%」）
- [ ] 申請〜入金のステップが**フロー図**で示されている
- [ ] 一次情報の**公式URLが明記**されている
- [ ] 読後に「次に何をすべきか」が**一言で**書かれている

### 安心感

- [ ] 制度の主体（国・都道府県など）が明記されている
- [ ] 怪しい断定（「必ずもらえる」「誰でも対象」）がない
- [ ] 「詳しくはお気軽にご相談を」など、相談への導線がある

### デザイン・実用性

- [ ] Lucide のみ（絵文字禁止）
- [ ] ADS 配色（`base.html` に沿う）
- [ ] **A4印刷・PDF出力で崩れない**
- [ ] スマホで縦読みしやすい
