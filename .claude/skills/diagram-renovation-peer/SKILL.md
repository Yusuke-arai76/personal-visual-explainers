# Diagram Renovation Peer

**窓・ドアのリフォーム業**のトピックを、**業界をほとんど知らない読者**が読んでも腹落ちし、**一人でお客様に説明できる**ところまで落とし込んだ図解HTMLを作る。

## 想定読者・成功定義（ヒアリング反映）

| 項目 | 内容 |
|------|------|
| 作者の主戦場 | 経営全般だがプレイヤーとしては**営業**に最も力を入れている |
| 読者 | **ADS（オンラインスクール）受講生の仲間**。リフォーム業界はほぼ未経験 |
| 典型ニーズ | 官公庁の**助成金・制度の文章だけでは追いづらい** → 図で流れ・関係が欲しい |
| 成功 | **知識ゼロの人**が、この図解付き説明を読んだあと**お客様へ説明できる** |

## 依存（このリポジトリ内）

- `.claude/skills/creating-visual-explainers/references/base.html` — 図解の**額縁**（Tailwind・Lucide・ADS配色・`CONTENT` プレースホルダー）
- `.claude/skills/creating-visual-explainers/references/model-answer.html` — 品質・パターンの参照
- `.claude/skills/creating-visual-explainers/scripts/deploy-diagram.sh` — surge.sh デプロイ
- `output/` — 生成HTMLの保存先

## ワークフロー

### Phase 0: リサーチ（サブエージェント）

**explore** サブエージェントで、制度・商品・工事の実務を調べ、図解に必要な事実を集める。

```
Task({
  subagent_type: "explore",
  description: "リフォーム・制度ドキュメントの調査",
  prompt: `
    以下について徹底的に調査してください：
    【対象】{URL or トピック名（例：〇〇助成金、断熱窓、建具交換の流れ）}

    調査項目：
    1. 一言で何の制度・何の工事・何の商材か（Why / 誰のためか）
    2. お客様・営業が迷いやすい用語一覧
    3. 典型的なシーン3つ以上（見積もり前、申請、施工当日など）
    4. 初心者が誤解しやすいポイント（条件・期限・対象外など）
    5. 公式一次情報のURL（省庁・自治体・業界団体など）

    結果は箇条書きで整理して返してください。
  `
})
```

### Phase 1: ソース読み込み

1. URLなら `WebFetch`、ファイルなら読み込み
2. **一次情報と噂を分ける**。図解に載せる条件・数値・期限は**出典が取れるもの**を優先
3. Phase 0 の結果と照合し、抜け漏れを潰す

### Phase 2: 用語翻訳表を作成

業界語・行政語を洗い出し、**仲間向けのやさしい言葉**と**お客様に言い換える例**を添える。

**詳細** → [references/term-dictionary.md](references/term-dictionary.md)

### Phase 3: たとえ話を3つ以上

| 種類 | 例（リフォーム文脈） |
|------|---------------------|
| 身近 | 銀行窓口、病院の受付、住宅の点検 |
| お金・手続き | クーポン、キャッシュバック、予約のキャンセル規定 |
| 時間軸 | カレンダー、駅の乗り換え、レストランのコース料理の順番 |

### Phase 4: 情報を絞り込む

- **「お客様説明で最初に言う3つ」**（または「まず覚える3つ」）を決める
- 全体像（どこに位置するか）を1枚の図または短いフローで示す
- 例外・細目は後半へ。読者が**その場で説明に使う順**に並べる

### Phase 5: 対話を設計

**キャラクター画像は使わない**。Lucide アイコン＋吹き出しで、**仲間（読者代弁）**と**営業の解説役**の対話を置く。

**詳細** → [references/character-usage.md](references/character-usage.md)

### Phase 6: HTML生成

1. `creating-visual-explainers/references/base.html` を `output/{スラッグ}.html` にコピー
2. `<!-- TITLE -->` / `<!-- DESCRIPTION -->` / `<!-- CONTENT_START -->`〜`<!-- CONTENT_END -->` を置換
3. Tailwind（`ads.*` 配色）と **Lucide のみ**（絵文字禁止）
4. 助成金・制度は **フロー図・条件分岐・タイムライン**を優先して視覚化
5. 一次情報URLは本文またはフッター近くに**出典として明記**

**詳細** → [references/html-structure.md](references/html-structure.md)

### Phase 7: 批判的レビュー（サブエージェント）

**ADS受講生でリフォーム未経験**の視点で忖度なくレビューする。

```
Task({
  subagent_type: "generalPurpose",
  description: "図解の批判的レビュー（業界未経験者）",
  readonly: true,
  prompt: `
    あなたは「リフォーム業界は初めてのADS受講生」です。
    専門用語は日常では使いません。お客様に説明した経験もほぼありません。

    以下のHTML図解を読んで、忖度なく批判してください：

    【HTMLファイルパス】{path}

    ## レビュー観点

    1. **わからない言葉**: 解説があっても難しい語はどれか
    2. **たとえの違和感**: ピンとこない・誤解を招くたとえはないか
    3. **情報過多**: 読むのが辛い、スキップしたくなる部分
    4. **対話の不自然さ**: 営業の現場感がないセリフはないか
    5. **お客様説明テスト**: 読み終えたあと、隣の人に1分で説明できる自信があるか
    6. **制度・数値の信頼性**: 出典がない断定がないか

    ## 出力形式

    ### 致命的な問題
    - ...

    ### 改善すべき点
    - ...

    ### 良かった点
    - ...

    ### 総評
    5段階評価と一言
  `
})
```

### Phase 8: ブラッシュアップ

Phase 7 を反映して HTML を修正する。

**優先度**: 致命的 → 改善 → 良かった点は維持

**よくある修正**:

- 用語ボックスの追加、お客様向け言い換え例の追加
- 助成金は**申請〜入金（または工事完了）**のタイムラインを足す
- 長文をカード分割、条件分岐を図示

### Phase 9: デプロイ

`creating-visual-explainers` の手順に従い、`deploy-diagram.sh` で surge.sh に公開する。

```bash
node --version   # 必要なら node-install-guide を参照
bash .claude/skills/creating-visual-explainers/scripts/deploy-diagram.sh output/{スラッグ}.html [スラッグ]
```

## 品質チェックリスト

### 必須

- [ ] 専門用語・行政語に**用語解説**（初出で必ず）
- [ ] たとえ話が**3つ以上**
- [ ] 導入・中間・まとめに**対話**（仲間 × 営業解説役）
- [ ] **「お客様説明で最初に言う3つ」**（または同等の絞り込み）がある
- [ ] 制度・数値・期限は**可能な範囲で出典URL**を付ける
- [ ] 読後、**お客様に渡せる説明の骨子**が一文またはチェックリストで残る

### デザイン

- [ ] Lucide のみ（絵文字禁止）
- [ ] `base.html` の ADS 配色・フォントに沿う
- [ ] スマホで読みやすい（`max-w-3xl` 想定内のレイアウト）
